The framework's GET /_agent-native/secrets list handler returns this payload shape
(from node_modules/@agent-native/core/src/secrets/routes.ts — DO NOT edit that file):

```typescript
interface SecretStatusPayload {
  key: string;
  label: string;
  description?: string;
  docsUrl?: string;
  scope: "user" | "org" | "workspace";
  kind: "api-key" | "oauth";
  required: boolean;
  status: "set" | "unset" | "invalid";
  last4?: string;      // only when status === "set" for api-key kind
  updatedAt?: number;  // ms, only when status === "set"
  oauthProvider?: string;
  oauthConnectUrl?: string;
  error?: string;
}
```

The client (node_modules/@agent-native/core/dist/client/settings/SecretsSection.js) just does
`fetch("/_agent-native/secrets")` → expects `SecretStatusPayload[]`. It keys off `status === "set"`
and renders `last4` if present. Matching the array shape is all that is required.

The registry of studio secrets (all `scope: "user"`, `kind: "api-key"` in register-secrets.ts):

```typescript
import { listRequiredSecrets } from "@agent-native/core/secrets";
// listRequiredSecrets(): RegisteredSecret[]  — { key, label, description?, docsUrl?, scope, kind, required?, oauthProvider?, oauthConnectUrl?, validator? }
```

The studio runtime reader to mirror (apps/staff-web/server/lib/app-secrets.ts):

```typescript
export async function readAppSecretByKey(key: string): Promise<string | null>;
// resolve-by-key (ignores scope/scope_id), decrypts, returns null on any failure.
```

How core mounts the route it must shadow (node_modules/@agent-native/core/src/server/core-routes-plugin.ts ~line 2210):
`getH3App(nitroApp).use("/_agent-native/secrets", handler)` → appends to `nitroApp.h3["~middleware"]`.
A handler that RETURNS a non-undefined result short-circuits the chain (next() is never called).
So the override must run BEFORE core's handler in the `~middleware` array → register via
`nitroApp.h3["~middleware"].unshift(...)` (the public `getH3App().use()` shim PUSHES, which would
land AFTER core and never win — do NOT use it for this).

registers a single middleware which intercepts ONLY the collection-level GET of
`/_agent-native/secrets` and rewrites api-key status to be studio-global (by key). Everything
else falls through to @agent-native/core unchanged.

Implementation requirements:

1. Import the studio-owned `register-secrets.ts` side-effect module at the TOP so the registry is
   populated even if this plugin loads before agent-chat.ts:
   `import "../register-secrets.js";`
   Also import:
   `import { listRequiredSecrets } from "@agent-native/core/secrets";`
   `import { readAppSecretByKey } from "../lib/app-secrets.js";`
   `import { defineNitroPlugin } from "nitropack/runtime"` (match the import path the other
   studio plugins use for defineNitroPlugin — check core-routes.ts / agent-chat.ts siblings; if
   they get it from "#imports" or a Nitro auto-import, follow that exact convention).

2. Extract a PURE, framework-free helper into the same file (exported for the unit test) — this is
   the bit Task 2 tests without a server:
   ```ts
   export interface SecretPresence { key: string; present: boolean; last4?: string }
   // builds the SecretStatusPayload[] from the registry + a presence map.
   export function buildSecretStatusPayload(
     secrets: { key: string; label: string; description?: string; docsUrl?: string;
                scope: "user"|"org"|"workspace"; kind: "api-key"|"oauth"; required?: boolean;
                oauthProvider?: string; oauthConnectUrl?: string }[],
     presenceByKey: Map<string, { present: boolean; last4?: string }>,
   ): /* SecretStatusPayload[] */ Array<Record<string, unknown>>
   ```
   Rules inside the helper:
   - For `kind === "oauth"`: return base with `status: "unset"` and pass through `oauthProvider` /
     `oauthConnectUrl` (we are NOT changing oauth behavior here — the override only fixes api-key
     presence; oauth status will be handled by core's own handler in the fall-through path… BUT
     since this override returns the whole array it must still emit oauth entries. Emit oauth
     entries with `status: "unset"` so they are never WORSE than before — note in a code comment
     that oauth presence is out of scope for this fix; in practice the studio registry has no
     oauth-kind secrets, so this branch is defensive only).
   - For `kind === "api-key"`: `status = present ? "set" : "unset"`; when present, set
     `last4` from the presence map. Do NOT set `updatedAt` (we resolve by key and don't carry a
     reliable per-row timestamp through `readAppSecretByKey`; omitting it is fine — the client
     only needs `status`. Leaving it undefined is acceptable).
   - NEVER put any plaintext value in the payload.

3. The middleware body (request-time):
   - Only handle the case where, after the mount prefix is stripped by core's mounting OR when
     matched directly, the request is **method GET** and the path is the collection root
     `/_agent-native/secrets` (no `:key`, no `/test`, no `/adhoc`). For ANY other method or any
     sub-path, call `next()` (return undefined) so core handles it. Match defensively on the FULL
     pathname: treat it as the collection root when the pathname ends with `/_agent-native/secrets`
     (allow an optional trailing slash) AND there is no segment after `secrets`. Reading
     `event.method` / `event.url.pathname` is the h3 v2 API used elsewhere in core.
   - When it IS the collection GET:
     a. `const secrets = listRequiredSecrets();`
     b. Build the presence map: for each api-key secret, `const v = await readAppSecretByKey(s.key);`
        then `presenceByKey.set(s.key, { present: v != null, last4: v ? "••••" + v.slice(-4) : undefined })`.
        (Compute last4 here — readAppSecretByKey returns the plaintext server-side; only the masked
        last4 leaves this function. Never log `v`.)
     c. `return buildSecretStatusPayload(secrets, presenceByKey);` — returning the array
        short-circuits the middleware chain so core's handler never runs for this request.
   - Wrap the whole thing so a failure NEVER 500s the panel: on any thrown error, `console.error`
     a redacted message and `return next()` to let core's handler answer (degrades to old behavior,
     never crashes).

4. Register the middleware so it runs BEFORE core's handler. Do NOT use `getH3App(nitroApp).use()`
   (that pushes to the end of `~middleware`, landing after core → it would never win because core
   returns a value and short-circuits). Instead unshift directly:
   ```ts
   const h3 = (nitroApp as any).h3;
   if (h3 && Array.isArray(h3["~middleware"])) {
     h3["~middleware"].unshift(theMiddlewareEntry);
   }
   ```
   Match the exact middleware-entry shape Nitro/h3 expects in this version — inspect one existing
   entry at runtime is not possible here, so mirror the shape core uses in
   node_modules/@agent-native/core/src/server/framework-request-handler.ts `registerMiddleware`
   (it pushes an object/handler onto `~middleware`; replicate that same shape, just via unshift).
   If the precise entry shape is a `{ route, handler }` object vs a bare function, follow whatever
   `registerMiddleware` constructs. Add a one-line comment explaining WHY unshift (precedence over
   core's pushed handler).

5. Header comment must state: (a) this is the studio-side fix for the by-key vs by-scope display
   bug; (b) it ONLY overrides the collection GET — writes/test/delete/oauth/ad-hoc stay in core;
   (c) it mirrors `readAppSecretByKey` (single-tenant: one Neon DB per studio); (d) it never edits
   @agent-native/core in-place (root AGENTS.md).
