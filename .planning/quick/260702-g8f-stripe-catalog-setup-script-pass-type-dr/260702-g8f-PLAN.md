<documented_facts>
These three facts were verified from the code and MUST be honored (reviewer to confirm before execution):

(a) HOW UNLIMITED IS REPRESENTED IN `passes` TODAY
    - There is NO null / infinity sentinel. Unlimited = `granted = 999` (a large int) with
      `expires_at` = the subscription's `current_period_end` (next billing date).
      Source: `apps/staff-web/server/db/seeds/seed-demo-data.ts:429-446`
      (`granted: 999, source: 'subscription', productName: 'Monthly Unlimited', expiresAt: periodEnd`).
    - Booking (`api.m.bookings.tsx:220`) computes `remaining = granted − SUM(its own debits)`; 999 is
      large enough that a monthly-unlimited pass never depletes within a billing cycle, and the
      period-end expiry retires it when the cycle ends.
    - THE NEW REDUCER MUST REUSE `granted = 999` for `credits === null` pass types. Do NOT invent a
      new sentinel — that would break the booking balance math.

(b) EXACT CURRENT IDEMPOTENCY MECHANISM IN THE REDUCERS
    - Every reducer runs inside ONE Drizzle transaction (WEB-06) and REFETCHES the object from Stripe
      (payload = notification only).
    - checkout.session.completed: passes granted with a DETERMINISTIC id
      `pass_<paymentIntentId>_<lineItemId>` + `ON CONFLICT (id) DO NOTHING`; payments id
      `pay_<paymentIntentId>` + `ON CONFLICT (stripe_payment_intent_id) DO NOTHING`;
      stripe_customers `ON CONFLICT (stripe_customer_id) DO NOTHING`.
    - invoice.paid: stripe_subscriptions `ON CONFLICT (stripe_subscription_id) DO UPDATE` (last-write-wins);
      payments `ON CONFLICT (stripe_payment_intent_id) DO NOTHING`. Currently grants NO pass.
    - THE NEW GRANTS MUST MIRROR THIS: deterministic pass id + `ON CONFLICT (id) DO NOTHING`.
      checkout keeps `pass_<pi>_<li>`; invoice adds `pass_sub_<invoiceId>` (one grant per invoice).

(c) DOES invoice.paid ALREADY FIRE ON RENEWALS?
    - YES. Stripe emits `invoice.paid` for EVERY paid invoice — the first subscription invoice AND every
      renewal. The current reducer already handles renewal semantics (falls back to `sub.metadata.memberId`,
      keys the CAPI event on the unique invoice id so each renewal reports; replay reuses the id → dedupe).
    - BUT it grants NO pass on any invoice today. The change is to ADD a pass grant keyed on the invoice id,
      which automatically grants once per cycle. Subscription grants live ONLY in invoice.paid — NOT in
      checkout.session.completed (which also fires for subscription checkouts) — to avoid double-granting.
</documented_facts>
