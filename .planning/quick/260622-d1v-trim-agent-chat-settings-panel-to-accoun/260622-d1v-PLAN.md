STEP A — Add a presence reader to `packages/core/src/secrets/storage.ts`.
Add a new exported async function near `listAppSecretsForScope` (after line ~349). It reports whether a key exists in `app_secrets` for the current studio (single Neon DB per studio), resolving STUDIO-GLOBAL by key alone (matching the runtime resolver `readAppSecretByKey`) — NOT scoped to the session email. It is presence-only: it MUST NOT decrypt or return the plaintext value.

```typescript
/**
 * Presence check: does an `app_secrets` row exist for this key, in ANY scope?
 *
 * Studio-global by design — one Neon DB per studio (single-tenant code,
 * multi-tenant deploy), so a key saved by any staff login counts as
 * configured for the whole studio. Mirrors the runtime `readAppSecretByKey`
 * resolution used by senders/workers. Presence-only: never decrypts or
 * returns the value, so it's safe to call from a status endpoint.
 */
export async function appSecretExistsByKey(key: string): Promise<boolean> {
  if (!key) return false;
  await ensureTable();
  const client = getDbExec();
  const { rows } = await client.execute({
    sql: `SELECT 1 FROM app_secrets WHERE key = ? LIMIT 1`,
    args: [key],
  });
  return rows.length > 0;
}
```
Notes:
  - `app_secrets` is a framework secrets table (not an `ownableColumns()` table); `storage.ts` already runs many raw `SELECT ... FROM app_secrets` queries without access helpers, so `guard:no-unscoped-queries` does not apply here. No `// guard:allow-unscoped` marker is needed (and don't add one unless the guard actually flags this file).
  - `ensureTable`, `getDbExec` are already in scope in this file.
