<objective>
Build the CORE missed-session re-engagement campaign capability for GymClassOS WhatsApp. Three parts, sequenced so each lands before the next consumes it:

1. **Opt-in capture + opt-out (Part 1)** — auto-create a `whatsapp_opt_in` row on a member's first inbound WhatsApp message; add a strictly-additive opt-out write path; make the worker `optInGate.ts` chokepoint honor opt-out.
2. **Batch send action (Part 2)** — `send-template-to-members` defineAction that fans out one `enqueueOutboundWhatsApp` job per member through the EXISTING queue → worker → `@gymos/whatsapp` chokepoint. No Meta calls, no `@gymos/whatsapp` import in staff-web.
3. **Campaign UI (Part 3)** — a focused `/gymos/campaigns` route where a coach picks the at-risk segment (reusing `list-at-risk-members`), picks an approved template (reusing TemplatesDialog patterns), sees an eligible-recipient preview count, confirms via AlertDialog, and sends.

Purpose: Coaches can re-engage missed-session members in bulk from one surface without leaving compliance gates (opt-in / 24h-window / template-approved) to the UI — the worker chokepoint stays authoritative.

Output: 1 additive migration + schema/mirror updates, opt-in auto-capture + opt-out gate, a batch fan-out action, and a minimal campaigns route with a nav entry point.

**SCOPE BOUNDARIES (explicit deferrals — do NOT build):**
- Scheduled/cron campaign automation — DEFERRED.
- Custom segment builder — DEFERRED; reuse `list-at-risk-members` only.
- "STOP"-keyword auto-detection from inbound text — DEFERRED (Task 1 provides the opt-out WRITE PATH + gate; keyword auto-detection is documented as deferred unless trivially cheap — see Task 1 action).
- Per-member variable personalisation in the batch send — out of scope; the batch action accepts a single shared `variables` map applied to all recipients (document this).
</objective>
