# Quick Task 260604-op8: inbound messages not stored / not in inbox

## Problem (evidence)

First real inbound WhatsApp (MYÜTIK relay) reached the worker but failed. pg-boss
`inbound-whatsapp` jobs at 16:34:12 and 16:34:55 → `state=failed, retry_count=5`,
`output` error `42P10: there is no unique or exclusion constraint matching the
ON CONFLICT specification`.

`services/worker/src/domain/conversations.ts` inserts the message with
`.onConflictDoNothing({ target: schema.messages.externalId })` → SQL
`ON CONFLICT (external_id) DO NOTHING`. But the unique index is **partial**:
`CREATE UNIQUE INDEX ... ON messages (external_id) WHERE (external_id IS NOT NULL)`.
Postgres cannot infer a partial index without the matching predicate → throws
42P10 on every member-matched inbound. The job's step 2 (conversation counter
bump) commits first, so retries inflate `unread_count` (saw 12) while **zero**
message rows are written.

Second issue (visibility): the test number maps to a `status='lead'` conversation
(created by the 06-02 Schedule Enquiry form). The inbox loader
(`apps/staff-web/app/routes/gymos.inbox.tsx`) defaults to `ne(status,'lead')`, so
leads never appear in the main inbox. **Decision (locked): promote to inbox** —
any inbound sets `status='open'`.

(Noted, not fixed here: two identical partial unique indexes on `external_id`
— `idx_messages_external_id` + `messages_external_id_unique` — baseline/migration
drift; harmless to this fix, prune later.)

## Fix

### Task 1: partial-index ON CONFLICT + promote-on-inbound

**Files:** `services/worker/src/domain/conversations.ts`

1. Import `sql` from `drizzle-orm` (alongside `eq, and`).
2. Message insert: add the partial-index predicate so ON CONFLICT can infer the
   index:
   ```ts
   .onConflictDoNothing({
     target: schema.messages.externalId,
     where: sql`${schema.messages.externalId} is not null`,
   })
   ```
   (Drizzle 0.45.2 `onConflictDoNothing({ target, where })` → `ON CONFLICT (col)
   WHERE <where> DO NOTHING`.)
3. Existing-conversation update (step 2 `.set({...})`): add `status: "open"` so
   any inbound promotes a `lead` (and reactivates `closed`/`snoozed`) into the
   working inbox. The new-conversation branch already sets `status:"open"`.

**Verify:** `cd services/worker && npx tsc --noEmit` clean. Generated SQL includes
the `WHERE external_id is not null` predicate (code review).

**Done:** member-matched inbound inserts a `messages` row and the conversation
shows `status='open'`; no 42P10.
