**create-class-occurrence.ts** (AES-05, reusable by AE2):
- description: "Schedule a class occurrence from an existing class definition. Resolves the definition for default capacity + duration; computes endsAt = startsAt + durationMin. Returns { id, startsAt, endsAt, capacity }."
- schema: z.object({
    definitionId: z.string().min(1),
    startsAt: z.string().min(1).describe("ISO datetime, studio-local with tz offset"),
    capacity: z.number().int().min(1).max(500).optional(),
    room: z.string().max(120).optional(),
    instructorUserId: z.string().optional(),
    notes: z.string().max(2000).optional(),
  })
- run:
  1. Validate `startsAt` parses: `const start = new Date(startsAt); if (isNaN(start.getTime())) return { error: "INVALID_STARTS_AT" };`
  2. Resolve the definition (must exist + be usable). `// guard:allow-unscoped` then select { id, durationMin, defaultCapacity } from schema.classDefinitions where eq(id, definitionId) limit 1. If none: `return { error: "DEFINITION_NOT_FOUND" };`
  3. Compute capacity = `capacity ?? def.defaultCapacity`. (Zod already guards capacity >= 1 when provided; defaultCapacity is >= 1 by its NOT NULL default 12.)
  4. Compute endsAt = `addMinutes(start, def.durationMin).toISOString()`.
  5. `// guard:allow-unscoped` insert schema.classOccurrences: id `cocc_${nanoid()}`, definitionId, startsAt (pass through the input string verbatim — do NOT re-serialise to UTC; studio-local ISO is stored as-is per the route's day-bucketing comment), endsAt, capacity, room (or null), instructorUserId (or null), notes (or null), status: "scheduled", createdAt: new Date().toISOString().
  6. Return `{ id, startsAt, endsAt, capacity }`.

Do NOT accept an inline newDefinition object — the UI orchestrates the two-step (cleaner + keeps each action atomic and agent-reusable). Document this choice in a top-of-file comment in create-class-occurrence.ts.

NOTE on endsAt vs UTC: `addMinutes` operates on the Date instant, then `.toISOString()` returns a Z (UTC) string. The route stores startsAt as the input string and buckets via `new Date(iso).toISOString().slice(0,10)`. Pass startsAt THROUGH unchanged; compute endsAt from the parsed instant. Add a comment documenting that endsAt is serialised in UTC while startsAt is passed through as the caller's studio-local ISO — both render correctly through `new Date(iso)` in the calendar. This is consistent with the existing demo UTC-bucket behaviour (SCH-07 production TZ work is out of scope).
