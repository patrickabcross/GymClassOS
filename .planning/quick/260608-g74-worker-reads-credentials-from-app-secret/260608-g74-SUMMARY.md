# Quick Task 260608-g74: Worker Reads Credentials from app_secrets

**One-liner:** Worker resolvers now prefer credentials stored in the framework `app_secrets` table (written by the staff-web Settings UI) over the pgcrypto `secrets` table and env vars, using a local AES-256-GCM reader that mirrors the framework's key derivation exactly.

## What Was Built

### Task 1: `readAppSecretByKey` (new file)

`services/worker/src/lib/appSecrets.ts` exports a single async function that:

1. Short-circuits to `null` when neither `SECRETS_ENCRYPTION_KEY` nor `BETTER_AUTH_SECRET` is set (emits a one-time debug log, does NOT query the DB).
2. Derives a 32-byte AES key via `sha256(material)` â€” identical to `getEncryptionKey()` in `packages/core/src/secrets/storage.ts`.
3. Queries `app_secrets` by key alone (single-tenant, no scope filter; `guard:allow-unscoped` comment).
4. Decrypts the `v1:<iv>:<ct>:<tag>` payload with AES-256-GCM, returning the plaintext.
5. Returns `null` (never throws) on any decrypt failure â€” bad prefix, missing parts, wrong key, tampered GCM tag.

`services/worker/src/lib/appSecrets.test.ts` covers 7 cases:

| Case | Outcome |
|---|---|
| Happy path (correct key + row) | Returns plaintext |
| Empty rows | `null` |
| No key material | `null`, DB not queried |
| Garbage `encrypted_value` | `null`, no throw |
| Wrong key (GCM tag fails) | `null`, no throw |
| Missing `v1:` prefix | `null`, no throw |
| `SECRETS_ENCRYPTION_KEY` takes precedence over `BETTER_AUTH_SECRET` | Returns plaintext |

## Deploy Note

**To activate this feature in production:**

Set the shared `BETTER_AUTH_SECRET` as a Fly secret on the worker app, using the **same value** that staff-web uses:

```bash
fly secrets set BETTER_AUTH_SECRET=<same-value-as-staff-web> -a <worker-app-name>
fly deploy --config services/worker/fly.toml ...
```

Until that secret is set, **every resolver falls back to its current behavior** (pgcrypto `secrets` table â†’ env). There is zero breakage risk on the current deploy.

If staff-web is later switched to a dedicated `SECRETS_ENCRYPTION_KEY` (separate from `BETTER_AUTH_SECRET`), set that on the worker instead â€” the worker checks `SECRETS_ENCRYPTION_KEY` first, exactly matching the framework precedence.
