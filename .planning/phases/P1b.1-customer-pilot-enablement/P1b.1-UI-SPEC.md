## Spacing Scale

Declared values (multiples of 4 only). These match the Tailwind v4 default scale and are already in use across existing gymos routes.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (`gap-1`), badge padding, inline icon margins |
| sm | 8px | Compact element spacing (`gap-2`), button horizontal rhythm |
| md | 16px | Default panel padding (`px-4`), form field spacing |
| lg | 24px | Section spacing within panels (`space-y-6`), dialog content padding |
| xl | 32px | Layout region gaps, page-level outer padding |
| 2xl | 48px | Major section breaks (analytics card grid rows) |
| 3xl | 64px | Full-page empty states, hero content blocks |

Exceptions:
- Nav bar height: 44px (`h-11`) — matches existing `GymosTopNav` exactly; do not change
- Nav tab padding: `px-2.5 py-1` (10px / 4px) — preserve existing tab rhythm
- Reply form area: `px-5 py-3` (20px / 12px) — matches existing form chrome in `gymos._index.tsx:522`
- Touch target minimum: 44px height for all interactive elements (buttons, nav tabs)
- Member context panel width: 300px fixed (`w-[300px]`) — matches existing right-rail

## Color

All values are CSS custom properties resolved from `apps/staff-web/app/global.css`. Use semantic tokens — never raw hex.

| Role | Token | Light mode value | Usage |
|------|-------|-----------------|-------|
| Dominant (60%) | `--background` / `bg-background` | `hsl(0 0% 100%)` = white | Page background, full-bleed surfaces |
| Secondary (30%) | `--card` / `bg-card` + `--sidebar-background` | `hsl(0 0% 100%)` + `hsl(220 10% 97%)` | Conversation list panel, member context panel, analytics cards (`bg-card/20`, `bg-card/40`) |
| Accent (10%) | `--accent` / `bg-accent` | `hsl(220 10% 95%)` = very light slate | Active nav tab background, hover states on nav tabs |
| Muted foreground | `--muted-foreground` | `hsl(220 5% 45%)` | All secondary labels, timestamps, captions, disabled placeholder text |
| Destructive | `--destructive` | `hsl(0 84% 60%)` = red | Failed-send error bubble text, destructive confirmation buttons only |
| Border | `--border` | `hsl(220 10% 90%)` | All panel dividers (`border-border/50` for softer variant) |

Accent reserved for:
- Active navigation tab background only (`bg-accent` on the selected tab in `GymosTopNav`)
- Nav tab hover state (`hover:bg-accent/40`)

Semantic colour additions for this phase:

| Semantic | Token | Usage |
|----------|-------|-------|
| Window open (in-window) | `text-emerald-600 dark:text-emerald-400` | WhatsApp 24h-window "in window" badge dot and label — matches existing pattern at `gymos._index.tsx:411` |

Dark mode: `global.css` dark tokens already defined. All new components must use semantic tokens so dark mode is automatic. Never hardcode light-only colors.

## Surface Specifications

### 1. Bare Gymos Layout (D-01)

**What changes:** `AppLayout.tsx` gains an `isGymosPath()` branch that short-circuits to `AgentSidebar`-only wrapper for `/gymos/*`. The email `AppLayoutInner` (hamburger, Important/Other tabs, email sidebar, Compose pen, refresh bell) is never rendered for gymos paths.

**Visual contract:**
- Full-width content area flush to the left edge (no 240px email sidebar)
- `GymosTopNav` (44px `h-11`) sits at the very top of the content area, below the AgentSidebar toggle strip
- Right-rail `AgentSidebar` with `defaultOpen={!isMobile}` — matches existing behavior at `AppLayout.tsx:139-150`
- `emptyStateText="Ask me anything about your gym"` — exact string, preserve casing
- Suggestions array: `["Provide renewal numbers", "Which classes haven't been filled in the last week?", "Which customers should I reach out to?"]` — exact strings, do not paraphrase

**Interaction:** No change to AgentSidebar toggle behavior. No change to `GymosTopNav` visual design — only the wrapper above it changes.

### 2. Templates Button + Picker Dialog (D-03, D-04, D-05)

**Button placement:** Secondary button inside the existing reply form's `flex gap-2` button row at `gymos._index.tsx:528`. Order (left to right): `Input` (flex-1) → `Templates` button → `Send` button.

**Button visual:**
- Label: "Templates"
- Icon: `IconTemplate` from `@tabler/icons-react` (size 14, `aria-hidden`, left of label text)
- Variant: `outline` (secondary to the primary `Send` button which uses default variant)
- Always enabled (regardless of window state — templates work both in and out of window)
- Width: auto / content-width

**Dialog layout (D-04):**
```
DialogContent — max-width: 640px, height: 520px, flex row
├── Left pane (200px fixed, border-r border-border/50)
│   ├── pane header: "Templates" heading (14px semibold) + subtitle "WhatsApp approved" (11px muted)
│   └── ScrollArea (flex-1)
│       └── template list rows (one per whatsapp_templates row)
│           ├── template name (13px regular)
│           ├── status badge: "approved" (green outline) or "awaiting approval" (muted, disabled row)
│           └── hover: bg-accent/40; selected: bg-accent
└── Right pane (flex-1, px-4 py-3)
    ├── [when no template selected] Empty state: "Select a template" (13px muted, centered)
    ├── [when template selected]
    │   ├── Template name (14px semibold)
    │   ├── Variable inputs — one Input per {{variable}} in components_json (label 12px, input 13px)
    │   ├── Separator (my-3)
    │   ├── Body preview section (12px label "Preview" uppercase tracking-wide muted)
    │   └── Preview block (13px, bg-muted/40 rounded p-3, line-height 1.5, monospace-free)
    └── Dialog footer (border-t border-border/50 px-4 py-3 flex justify-end gap-2)
        ├── Cancel button (variant=ghost)
        └── Send button (variant=default, disabled if no approved template selected or required variables empty)
```

**Interaction states:**
- Selecting a template row: highlights row (`bg-accent`), populates right pane, clears any prior variable inputs
- Variable inputs: standard `Input` components; validation is presence-only (no empty required fields) — error shown inline below input (`text-[11px] text-destructive`)
- Preview updates live as variables are typed (client-side string substitution of `{{variable_name}}` placeholders)
- Send: calls the existing `enqueueOutboundWhatsApp` action; on success closes dialog + shows Sonner toast "Template queued" (green); on error shows Sonner toast with the typed error code copy
- Cancel: closes dialog, discards state
- Keyboard: `Escape` closes dialog (shadcn default); `Tab` moves through template list then variable inputs then buttons

**Template list rows for seeded templates:**
- `hello_world` — status badge: "approved" (enabled row)
- `class_reminder`, `waitlist_offer`, `payment_failed`, `pass_expiring` — status badge: "awaiting approval" (row disabled via `opacity-50 cursor-not-allowed`, wrapped in `Tooltip` with content "Awaiting Meta approval — submit templates in P0/FND-06")

### 3. Analytics Route `/gymos/analytics` (D-Claude's Discretion)

**Nav tab:** Add "Analytics" `Link` between `Payments` and `Settings` in `GymosTopNav.tsx`. Same `tabClass()` pattern. No icon (consistent with other text-only tabs). Tab text: "Analytics" (exact, title case).

**Page layout:**
```
/gymos/analytics
└── div (flex flex-col gap-6 p-4)
    ├── Page header
    │   ├── h1: "Analytics" (14px semibold)
    │   └── Subtitle: "Last 7 days · Last 30 days" (11px muted, static labels — no date picker)
    ├── Metric card grid (grid grid-cols-3 gap-4)
    │   ├── Card: Fill Rate
    │   ├── Card: Cancellation Rate
    │   └── Card: Pass Utilisation
    └── [optional second row if planner adds a 4th metric]
        └── Card: No-show Rate (Claude's discretion to include or defer)
```

**Metric card anatomy (consistent across all 3+):**
```
Card (p-4 border border-border/50 rounded-lg bg-card/40)
├── CardHeader (pb-2)
│   ├── label: metric name (12px uppercase tracking-wide muted-foreground)
│   └── period badge: "7d" / "30d" (10px, bg-muted rounded-full px-1.5)
├── CardContent
│   ├── primary value: "73%" or "12" (28px, weight 600) — `text-2xl font-semibold`
│   ├── secondary context: "of 15 classes" or "4 cancellations" (12px muted)
│   └── [loading state]: Skeleton (h-8 w-24 + h-4 w-32)
```

Both 7-day and 30-day values shown on same card (stacked, 7d primary, 30d secondary row). No date-range picker — defer to P2.

**Metrics to implement (minimum 3, planner verifies schema):**
1. **Fill Rate**: `SUM(bookings) / SUM(capacity)` over `class_occurrences` × `bookings` for the trailing 7 / 30 days
2. **Cancellation Rate**: cancelled bookings / total bookings (trailing 7 / 30 days)
3. **Pass Utilisation**: active passes with at least 1 debit / total active passes (snapshot)

Empty state (zero seeded data): show cards with "–" as the value and "No data yet" as context. Do not hide the cards.

### 4. Customer Login / Access Denied Page (D-06, D-07)

**`/access-denied` page:**
- Full-page centered layout (`min-h-screen flex flex-col items-center justify-center gap-6 bg-background`)

**Sign-in page (no design work needed):** Better-auth Google OAuth flow uses the existing sign-in page. Only the redirect-after-sign-in target and the allowlist check are new behavior.

### 5. Gym-Aware Agent Surface (D-08, D-09, D-10)

**No UI changes** — the AgentSidebar component is unchanged. Visual contract is maintained by the existing sidebar. The design contract for the chip prompts:
- Suggestion chips render as plain text buttons inside AgentSidebar (framework-controlled rendering)
- Exact chip copy: `"Provide renewal numbers"`, `"Which classes haven't been filled in the last week?"`, `"Which customers should I reach out to?"` — preserve verbatim, sentence case
- Agent responses are plain text / markdown; no special card components needed in P1b.1 (read-only data, prose answers acceptable)

## Interaction Contracts

### Optimistic UI (mandatory per CLAUDE.md)

All mutations use the optimistic-insert pattern already established at `gymos._index.tsx:299-301`:
- Template send: immediately inserts a `messages` row with `status='queued'` into TanStack Query cache; dialog closes; Sonner toast fires; worker processes asynchronously
- No spinner blocking the dialog close — dialog closes on Send button click, not on server acknowledgement

### Loading states

| Surface | Loading treatment |
|---------|------------------|
| Analytics metric cards | `Skeleton` (h-8 w-24 for value, h-4 w-32 for context) per card; grid is always visible |
| Template list in dialog | `Skeleton` rows (h-8 w-full, 3 rows) while `whatsapp_templates` loads |
| Analytics page first load | Skeleton cards shown immediately; data replaces them |

### Error states

| Error | Treatment |
|-------|-----------|
| Template send fails (typed error from worker) | Sonner toast (destructive variant) with human-readable copy from copywriting contract above; failed-bubble appears on the queued message row (existing `failedCopy(errorCode)` pattern) |
| Analytics data load fails | Each card shows "–" value + "Failed to load" in muted 11px text; no full-page error |
| Access denied (allowlist miss) | Redirect to `/access-denied` page (see surface spec above) |
| Template variable missing on Send attempt | Inline `text-[11px] text-destructive` below the empty required input; Send button stays disabled |

### Keyboard and accessibility

- All Dialog interactions: `Escape` closes, `Tab` navigates through template list → variable inputs → footer buttons
- Template list rows: `role="listbox"` + `role="option"` + `aria-selected` (shadcn ScrollArea + explicit ARIA)
- Analytics cards: `role="region"` + `aria-label="{metric name}"` per card
- Nav tab for Analytics: same `<Link>` pattern as existing tabs; no ARIA additions beyond what exists
- Access denied page: `role="main"`, `lang` inherited from root, no special ARIA
