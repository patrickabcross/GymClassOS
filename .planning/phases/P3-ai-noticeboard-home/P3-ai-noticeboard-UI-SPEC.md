## Resolved Aesthetic Direction

### The Noticeboard vs. Clean-Minimal Tension — Resolved

The "Polsia-style noticeboard" aesthetic is achieved through **layout metaphor and typographic hierarchy**, not decoration. This resolution is mandatory — do not introduce cork textures, pinned-tape images, drop-shadow theatrics, or skeuomorphic card styling.

**What "noticeboard" means in this spec:**
- A board surface with a slightly warm off-white background (`bg-[hsl(var(--muted)/0.4)]`) that reads as a surface you pin things to, not a blank app screen.
- Section cards that sit *on* the board as distinct pinned areas — achieved via `bg-card border border-border/60 shadow-sm rounded-lg`.
- Section labels that read as **bold bulletin headers** — `text-xs uppercase tracking-wider font-semibold text-muted-foreground` — like a physical section divider pinned above a group of notes.
- Priority indicators using a `border-l-4` left-border strip on task rows (a pinned-note visual cue), not colored badge blobs.
- Card content that feels handwritten/curated, not generated — AI body text in `text-sm text-foreground/80 italic` within an inset zone (`bg-muted/40 rounded-md p-3`), visually distinct from the metric subheading.

**What "clean/minimal" requires (AGENTS.md mandate — non-negotiable):**
- No competing controls: each card's primary action is the one CTA visible; secondary actions collapse into a `DropdownMenu` overflow (`...`).
- Progressive disclosure: the full proposal payload is behind the AlertDialog, not printed on the card.
- No always-visible toolbars. No decorative badges that don't carry information.
- The board chrome (background, section dividers) should vanish once the coach is reading content — it is infrastructure, not personality.

**Resolved rule:** The noticeboard character comes from the *spatial arrangement* and *tone of copy*, not from graphic embellishment. One off-white background layer, card borders at reduced opacity, and a left-border priority strip are the full extent of the "bulletin" aesthetic. Any PR that adds textures, gradient overlays, or shadow theatrics beyond `shadow-sm` violates this spec.

**Visual hierarchy:** The AiTodayStrip at the top of the board is the primary focal point and is read first; the four-card grid below it is read second. The TasksSection anchors the bottom of the board and is read last.

|-------|----------|-------|-------|

| md | `p-4` / `gap-4` | 16px | Default element spacing, card body padding |

| xl | `p-8` / `gap-8` | 32px | Board outer padding |
| 2xl | `p-12` / `gap-12` | 48px | Major section breaks (board → tasks) |

Exceptions:
- Task list item touch target minimum: `min-h-[44px]` (accessibility — one-click Approve must be easily tappable on a trackpad or tablet)
- Board card grid gap: `gap-4` (16px) between cards on all breakpoints — keeps the board feel tight, avoids cavernous whitespace

## Copywriting Contract

### Global patterns

| Element | Copy |
|---------|------|
| Primary CTA — approve proposal | "Approve" (verb only — the AlertDialog explains what will be approved) |
| Primary CTA — complete task | "Mark done" |
| Primary CTA — reject proposal | "Dismiss proposal" (softer than "Reject" — avoids unnecessary alarm for routine dismissals) |
| Destructive confirm — WhatsApp send | "Send [N] WhatsApp messages?" (AlertDialog title) |
| Destructive confirm body | "This will send [templateName] to [N] member[s]. Messages that are out of window or not opted-in will be skipped by the worker. This action cannot be undone." |
| Destructive confirm action button | "Send messages" |
| Destructive confirm cancel | "Cancel" |

### Section card copy patterns

| Card | Title label | Computed subheading pattern | Empty AI note |
|------|-------------|------------------------------|----------------|
| Inbox | "INBOX" | "[N] unread · [M] open conversations" | "No notes yet. Ask the agent to review your inbox." |
| Schedule | "SCHEDULE" | "[N]% avg fill rate this week · [M] classes" | "No notes yet. Ask the agent about upcoming classes." |
| Members | "MEMBERS" | "[N] active · [M] at risk of lapsing" | "No notes yet. Ask the agent about member retention." |
| Revenue | "REVENUE" | "£[MRR]/mo MRR · [N] net new this month" | "No notes yet. Ask the agent about revenue trends." |

**Rules for computed subheadings:**
- Always show a real value or a dash (`—`) when data is loading or unavailable — never show "Loading..." as persistent copy.
- "At risk" count comes from `list-at-risk-members` `.length` (default 14-day inactivity). Show as "[N] at risk" only when N > 0; otherwise show "no members at risk" (positive framing).
- Net growth: show as "+[N]", "−[N]", or "flat" — never just a number without direction.

### AI Today strip copy patterns

| State | Copy |
|-------|------|
| Idle (no `ai_today` note) | "The agent is ready. Ask a question or request a recommendation in the chat." |
| Has note | Show the note body verbatim (agent-authored; keep concise — `max(2000 chars)` enforced in `upsert-section-note` schema) |
| Active (agent is responding in chat, detected via `useDbSync` signal) | "Working…" prefix before the note, or pulse indicator |

### Empty states

| Surface | Heading | Body |
|---------|---------|------|
| Tasks — no tasks | "No tasks yet" | "The agent will create tasks here as it identifies recommendations. You can also create tasks by asking in the chat." |
| Board — no data at all (fresh deploy) | (section cards still render with skeleton metrics) | Each card shows its empty AI note copy (see above) |

### Error states

| Surface | Error copy | Recovery path |
|---------|------------|---------------|
| Metric fails to load (one `list-*` action errors) | Card subheading shows "—" in place of the metric | Tooltip on "—": "Metric unavailable. Refresh to retry." |
| Approve proposal fails | Sonner toast: "Action failed. [error message from server]. The proposal is still pending." | Proposal card remains visible; coach can retry |
| Complete task fails | Toast: "Could not mark task as done. Please try again." | Task reverts to uncompleted state (optimistic rollback) |

### `BoardCard`

**File:** `apps/staff-web/app/components/gymos/Noticeboard/BoardCard.tsx`

**Purpose:** One section card (Inbox / Schedule / Members / Revenue).

**Anatomy (top to bottom):**
```
CardHeader:
  [Section title label — 12px uppercase tracking-wider semibold muted]   [...overflow DropdownMenu trigger: aria-label="More options"]
  [Computed subheading — 14px regular (400) text-muted-foreground]

CardContent:
  [AI note inset zone — bg-muted/40 rounded-md p-3, only if note exists]
    [AI body text — 14px italic text-foreground/80]
    [note timestamp — text-xs text-muted-foreground mt-1]

  [Pending proposal zone — only if proposal for this section exists]
    [Separator]
    [Proposal rationale — text-sm text-foreground/80]
    [Approve button + Dismiss proposal button]
```

**Card sizing:** `min-h-[160px]` — cards have a minimum height to prevent jarring reflows during loading
**Card padding:** `p-4` (`CardHeader` + `CardContent` both use `p-4 pt-0` per shadcn default — do not override)
**Section title:** plain `<div className="text-xs uppercase tracking-wider font-semibold text-muted-foreground">` — NOT `<CardTitle>`
**Overflow menu:** `<DropdownMenu>` trigger is `<Button variant="ghost" size="icon" aria-label="More options">` with `IconDots` (16px). Contains: "Navigate to [section]" (calls `navigate` action), "Refresh" (invalidates the relevant `useActionQuery` key).
**Approve button:** `<Button variant="default" size="sm">` + `IconCheck` icon (14px) — triggers AlertDialog for outbound/irreversible proposals, or directly calls `approve-proposal` for navigate-type proposals (no AlertDialog needed for non-destructive).
**Dismiss proposal button:** `<Button variant="ghost" size="sm" className="text-muted-foreground">` + `IconX` icon (14px) — calls `reject-proposal` action optimistically.

**Note inset zone:** Only renders when `note.body` is non-empty. If empty, shows the empty AI note copy as `text-sm text-muted-foreground` (no inset background, no italic — distinguish "agent wrote nothing" from "agent wrote something").

**Proposal zone:** Only renders when a `dashboard_proposals` row with `status='pending'` is associated with this section. Association: Inbox card shows proposals where `actionName='send-template-to-members'`; Revenue card shows proposals where `actionName='create-checkout-link'`. Members card shows at-risk proposals (`send-template-to-members` targeting at-risk members).

- `priority === 2`: `border-l-amber-400`
- `priority === 3`: `border-l-border`

- Heading: `text-sm font-semibold text-muted-foreground` — "No tasks yet"
- Body: `text-xs text-muted-foreground mt-1` — "The agent will create tasks here…"

**No drag-to-reorder for V1.** Tasks render in SQL `ORDER BY priority ASC, created_at ASC` — sort is done server-side.

## Interaction Contracts

### Propose → Approve → Execute

This is the most consequential interaction in Phase P3. The spec is precise and binding.

#### For `send-template-to-members` proposals (outbound WhatsApp — irreversible):

1. Coach sees the proposal zone in the relevant BoardCard:
   - Rationale text (from `dashboard_proposals.rationale`) in `text-sm text-foreground/80`
   - "Approve" `Button variant="default" size="sm"` with `IconCheck` icon
   - "Dismiss proposal" `Button variant="ghost" size="sm"` with `IconX` icon

2. Coach clicks "Approve" → triggers `<AlertDialog>` (NOT a custom modal). The AlertDialog is the confirmation gate for irreversible outbound actions.

3. AlertDialog content (exact copy from the copywriting contract):
   ```
   AlertDialogHeader:
     AlertDialogTitle: "Send [N] WhatsApp messages?"
     AlertDialogDescription:
       "This will send [templateName] to [N] member[s].
       Messages that are out of window or not opted-in will be skipped
       by the worker. This action cannot be undone."

   AlertDialogFooter:
     AlertDialogCancel: "Cancel"
     AlertDialogAction: "Send messages"    ← destructive intent, styled with bg-primary (not --destructive; sending IS intended action)
   ```
   **Note:** The AlertDialog `description` section MUST show the template name and recipient count from the stored `paramsJson`. The executor must parse `paramsJson` before rendering the AlertDialog and pass these values as props.

4. Coach clicks "Send messages" → `approve-proposal` action fires (POST). Optimistic update: proposal card shows a loading state (`opacity-50`, Approve button disabled, spinner on button).

5. On success: proposal zone disappears from the card (optimistic removal). Toast: "Sent to [N] members." (`sonner` toast, variant info).

6. On failure: proposal zone snaps back. Toast: error copy from copywriting contract.

#### For `create-checkout-link` proposals (Stripe Checkout — reversible link generation):

No AlertDialog required — generating a Checkout link is not irreversible. Coach clicks "Approve" → direct `approve-proposal` call → Toast: "Checkout link ready. Copied to clipboard." The resulting URL is copied to clipboard in the success handler.

#### For navigate proposals:

No confirmation required. Clicks trigger the `navigate` action directly.

### Optimistic UI — mandatory for all mutations

| Action | Optimistic update | Rollback on error |
|--------|-------------------|-------------------|
| Complete task | Immediate `opacity-50 line-through` + `IconCircleCheck` swap | Restore full opacity + `IconCircle` + error toast |
| Dismiss proposal | Proposal zone immediately collapses from card | Restore proposal zone + error toast |
| Approve proposal (send) | Proposal zone shows loading state, button disabled | Restore enabled state + error toast |
| Approve proposal (checkout) | Button shows loading state | Restore + error toast |

All optimistic updates use React Query's `onMutate` / `onError` / `onSuccess` pattern via `useActionMutation` (the existing hook in staff-web).

### Loading States — Skeleton contract

The noticeboard uses a hybrid loading pattern: the route loader fetches persisted dashboard state (notes, tasks, proposals — fast, single-tenant DB query). Live metrics (`list-*` actions) fetch client-side via `useActionQuery`.

**Skeleton placement:** Each `BoardCard` renders a skeleton subheading while its metric query is loading.

```tsx
// Metric subheading in BoardCard
{isLoadingMetric ? (
  <Skeleton className="h-4 w-40 mt-1" />
) : (
  <div className="text-sm text-muted-foreground mt-1">{computedSubheading}</div>
)}
```

**Do NOT skeleton the AI note.** Notes come from the loader (server-rendered), so they are available on first paint. Show note content immediately.
**Do NOT skeleton the Tasks section.** Tasks also come from the loader.
**Do NOT show a full-page loading spinner.** The board structure is always visible; only metric values go skeleton.

---

### Error States — metric failure graceful degradation

If a `useActionQuery` call for a metric returns an error:

```tsx
{isErrorMetric ? (
  <Tooltip>
    <TooltipTrigger asChild>
      <span className="text-sm text-muted-foreground cursor-help">—</span>
    </TooltipTrigger>
    <TooltipContent>Metric unavailable. Refresh to retry.</TooltipContent>
  </Tooltip>
) : ...}
```

The card MUST remain rendered and readable even when metrics fail. The AI note and proposal zone are unaffected by metric fetch errors.

## Non-Blocking Open Questions

The following questions are documented but do NOT block implementation. The spec's defaults apply unless the user overrides.

1. **Should the `ai_today` strip show a "last updated" timestamp?**
   Default decision: No timestamp on the strip. The note body can include temporal language ("Just now, I sent win-back messages to 4 members…"). Adding a timestamp label creates visual noise. Override: add `text-xs text-muted-foreground` timestamp right-aligned if the agent's note is > 30 min old.

2. **Should completed tasks be shown with a strikethrough or hidden?**
   Default decision: Hide completed tasks from the tasks list (the query filters `status='open'`). A "Show completed" collapse/expand (`<Collapsible>`) can be added in P4 when the task list grows. Override: show all tasks with completed ones rendered at `opacity-40 line-through` at the bottom.

3. **Should the Revenue card show the primary metric as `text-2xl` or larger?**
   Default decision: `text-2xl font-semibold` (24px, 600) — consistent with the board's card sizing, smaller than analytics' `text-4xl` hero MRR which is a full-page metric. The noticeboard card is smaller than a full analytics dashboard; `text-2xl` is the right scale. Override: match analytics at `text-3xl` if the customer requests it.
